01 · Overview

SUBJECT
OVERVIEW

This document summarises the practical lab work completed throughout the UTS subject 48730/32548 — Networking & Cybersecurity in Autumn 2025. The labs covered a wide range of cybersecurity fundamentals, from hands-on packet analysis to configuring enterprise-grade network security on real Juniper SRX devices.

Each week built on the last — starting with passive information gathering, moving through active attacks, encryption, intrusion detection, and finishing with enterprise firewall configuration. The labs used both Linux-based virtual machines and physical network infrastructure accessible via SSH through the UTS network.

// Lab Environment

All VM-based labs ran inside a VMware environment with three VMs: Cybersec-Server (10.0.2.6), Cybersec-Client (10.0.2.8), and Cybersec-Attacker (10.0.2.7). Juniper labs connected through a boundary router at 138.25.62.61 on the UTS campus network.

02 · Tools & Environment

TOOLS &
ENVIRONMENT

The following tools and platforms were used across all lab sessions throughout the semester.

WIRESHARK
Packet Analysis
Open-source packet capture and protocol analyser. Used for sniffing HTTP traffic, capturing DNS packets, and observing TCP/IP attacks in real time.
NMAP / ZENMAP
Port Scanning
Network mapper used for aggressive port scans, OS detection, and service version discovery across target hosts.
NETCRAFT
OSINT / Recon
Web-based OSINT tool for gathering information about a target domain — hosting, IP, OS, web server, and DNS records.
JOHN THE RIPPER
Password Cracking
Password hash cracker supporting single, wordlist, and incremental (brute force) modes. Used against a Unix-style password file.
OPENSSL
Cryptography
Used for symmetric (AES) and asymmetric (RSA) encryption/decryption, generating key pairs, creating Certificate Authorities, and signing server certificates.
NETWAG
Attack Simulation
GUI frontend for Netwox packet crafting tools. Used to simulate TCP RST attacks, DNS spoofing, ICMP floods, and TCP session hijacking.
SNORT
IDS / IPS
Open-source network intrusion detection and prevention system. Configured custom SPL rules, run in IDS mode and IPS mode to detect and reject network threats.
IPTABLES
Linux Firewall
Linux kernel firewall utility. Used to configure INPUT/OUTPUT chain rules to permit HTTP/HTTPS and drop all other traffic on the server.
JUNOS CLI
Enterprise Firewall
Juniper SRX operating system. Used to configure IDS screens, security zone policies, and pass-through firewall user authentication on real hardware.
DIG / NSLOOKUP
DNS Tools
Command-line DNS query tools used to verify DNS resolution before and after pharming attacks, observing changes in answer, authority, and additional sections.
// key command patterns used across labs
# Packet capture
$ sudo wireshark
# Port scanning
$ nmap -A -p- <target-ip>
# Password cracking
$ john --wordlist=password.lst mypasswd
# Symmetric encryption
$ openssl enc -aes-128-cbc -e -in file.txt -out Encrypted.enc -K <key> -iv 0000
# Snort IDS live console mode
$ sudo snort -A console -q -c /etc/snort/snort.conf -i eth0
# iptables — drop all non-HTTP/S
$ sudo iptables -A INPUT -p tcp --dport 80 -m state --state NEW,ESTABLISHED -j ACCEPT
03 · Lab Summaries

LAB
SUMMARIES

Week 2 · Lab 1 Part 1

INFORMATION
GATHERING

The first lab introduced passive and active information gathering techniques. The core theme was understanding how unencrypted traffic exposes sensitive data, and how attackers can gather intelligence about a target before launching an attack.

// Task 1
Sniffing HTTP Login Credentials
Using Wireshark with promiscuous mode enabled, HTTP traffic was captured while logging into an intentionally vulnerable test website. By filtering for POST packets, the username and password transmitted in plaintext were clearly visible inside the HTML Form URL-encoded data of the packet. This demonstrated why HTTP (without HTTPS) is inherently insecure for transmitting credentials.
Wireshark HTTP Packet Filtering
// Task 2
Extracting Images from HTTP Traffic
Using the same captured packet session, the Export Objects → HTTP feature in Wireshark was used to extract image files (logo.gif) directly from the packet capture. This showed how any file transmitted over unencrypted HTTP can be fully reconstructed from captured traffic.
Wireshark HTTP Object Export
// Task 3
OSINT on uts.edu.au
Zenmap and NetCraft were used to gather public information about www.uts.edu.au. Findings included IP address, hosting company, operating system, web server type, DNS admin email, reverse DNS, domain registrar, and nameserver organisation. Typing the bare IP into a browser returned a Cloudflare Error 1003 — showing that direct IP access is blocked by the CDN. This task highlighted how much infrastructure information is publicly accessible.
NetCraft Zenmap OSINT
// Key Learning

Unencrypted HTTP transmits all data including credentials in plaintext. Any attacker on the same network can capture and read this data using nothing more than Wireshark. HTTPS is non-negotiable for any login or sensitive form.

Week 2–3 · Lab 1 Part 2

PASSWORD CRACKING
& SQL INJECTION

The second part of Lab 1 moved into offensive techniques using a local Cybersec-Server VM. The two main areas were cracking hashed passwords and exploiting SQL injection vulnerabilities in a login form.

// Task 4
Password Cracking with John the Ripper
A Unix-style password file (mypasswd) was cracked using John the Ripper's three modes. Single crack mode broke simple username-based passwords almost instantly. Wordlist mode used a dictionary (password.lst) to crack more complex ones. The --rules flag applied mangling transformations to the wordlist, eventually cracking even the most resistant password in the file. This reinforced how weak passwords — especially dictionary words — fall quickly against automated tools.
John the Ripper Hash Cracking Wordlist Attack Brute Force
// Task 5
SQL Injection — Login Bypass
A locally hosted PHP/MySQL login page was exploited using classic SQL injection. By entering a single quote ' in the username field, a MySQL syntax error was triggered, confirming the database type. The login was then bypassed by injecting an always-true SQL condition into the password field. Further exploitation dumped all usernames and passwords from the database to a file using SQL's outfile capability.
SQL Injection MySQL PHP Login Bypass
// Task 6
Port Scanning with Nmap
An aggressive (-A) port scan was run from the Attacker VM against both Cybersec-Server and Cybersec-Client using Zenmap's GUI. The scan revealed open ports including FTP (21), SSH (22), Telnet (23), DNS (53), and HTTP (80) on the server, along with running service versions. This demonstrated how an attacker maps a network before exploiting services.
Nmap Zenmap Service Detection OS Fingerprinting
Week 3 · Lab 2

DNS
PHARMING

This lab explored how DNS can be manipulated to redirect users to attacker-controlled addresses — a technique known as pharming. Three attack methods were demonstrated, each escalating in sophistication.

// Task 1
Attack via Interfaces File
The dns-nameserver entry in /etc/network/interfaces was changed to point to a rogue DNS server (10.0.2.7). After rebooting the network interface, the dig command confirmed that DNS queries were now being resolved by the attacker's server instead of the legitimate one — silently redirecting all hostname lookups.
dig nano DNS Hijacking
// Task 2
DNS Response Spoofing (Netwag Tool 105)
Netwag's Tool 105 — "Sniff and send DNS answers" — was configured on the Attacker VM to intercept DNS queries for a target hostname and inject a forged DNS response with a spoofed IP. Wireshark on the Client VM captured both the legitimate response and the spoofed response arriving almost simultaneously. This is a classic race condition attack on DNS.
Netwag (Tool 105) Wireshark DNS Spoofing
// Task 3
DNS Cache Poisoning
The same Netwag tool was used but now targeting the DNS cache on the server itself. After flushing the DNS cache with sudo rndc flush, repeated dig queries were sent for www.uts.edu.au while Tool 105 was running with a spoofed authoritative nameserver. Eventually the forged response was cached, and the authority and additional sections of the dig output reflected the attacker's nameserver.
Netwag (Tool 105) dig rndc Cache Poisoning
// Security Implication

DNS is fundamentally trust-based and unauthenticated in its basic form. These attacks show why DNSSEC exists — without cryptographic signing of DNS records, spoofed responses are indistinguishable from legitimate ones.

Week 4 · Lab 3

CRYPTOGRAPHY
& PKI

This was one of the most involved labs, covering the full lifecycle of cryptographic operations from basic symmetric encryption through to operating a Certificate Authority and serving HTTPS.

// Task 1
Symmetric Encryption with OpenSSL (AES-128-CBC)
A text file and a BMP image were encrypted and decrypted using AES-128-CBC via OpenSSL. The encrypted text file appeared as garbled characters when cat was run on it. Interestingly, encrypting the BMP image in ECB mode preserved some visual patterns (a known weakness of ECB), while CBC mode produced a fully scrambled result — demonstrating the practical difference between cipher modes.
OpenSSL AES-128-CBC Symmetric Encryption
// Task 2
Asymmetric Encryption (RSA)
An RSA key pair (2048-bit) was generated. A text file was encrypted with the public key and decrypted with the private key. The encrypted output was unreadable binary data. This demonstrated the core principle of public-key cryptography — only the private key holder can decrypt content encrypted with the corresponding public key.
OpenSSL RSA 2048-bit Public Key Crypto
// Task 3 & 4
Certificate Authority & Certificate Signing
A self-signed Certificate Authority (CA) was created using OpenSSL with organisation details (UTS/FEIT). A Certificate Signing Request (CSR) was then generated for cybersec.com.au and signed by the CA to produce a server certificate. This mirrors exactly how real-world TLS certificates are issued by trusted CAs.
OpenSSL X.509 CA / CSR PKI
// Task 5
Running an HTTPS Server
The server certificate and private key were combined into a PEM file and used to launch an OpenSSL HTTPS test server. After importing the CA certificate into Firefox's certificate store, browsing to https://cybersec.com.au:4433 showed a secure padlock — confirming the full PKI chain was working correctly.
OpenSSL s_server HTTPS / TLS PKI
Week 5 · Lab 4

TCP/IP
ATTACKS

This lab focused on active attacks against established TCP/IP connections — demonstrating how network sessions can be disrupted, hijacked, and manipulated at the packet level without any access to application-layer credentials.

// Task 4
TCP RST Attack on Telnet & SSH
Netwag Tool 78 — "Reset every TCP packet" — was run on the Attacker VM while a Telnet session was active between the Server and Client. Every time data was typed in the Telnet session, Wireshark captured RST/ACK packets arriving on the Client, causing the connection to drop with "Connection closed by foreign host." The same attack was repeated against an SSH session with identical results.
Netwag (Tool 78) Wireshark Telnet TCP RST
// Task 5a
ICMP Blind Connection-Reset
Netwag Tool 82 — "Sniff and send ICMP4 destination unreachable" — injected forged ICMP Type 3 Code 2 packets while the Client was pinging the Server. Wireshark showed the ping replies being overridden by "Destination Host Unreachable" messages from the attacker's spoofed IP, disrupting the ICMP connection stream.
Netwag (Tool 82) Wireshark ICMP Spoofing
// Task 5b
ICMP Source Quench Attack
Tool 85 sent forged ICMP Type 4 (Source Quench) messages, telling the TCP sender to slow down. While pinging, "Source Quench" messages appeared interleaved with echo replies in the Wireshark capture, simulating a congestion signal that could degrade or disrupt TCP throughput.
Netwag (Tool 85) ICMP Type 4
// Task 6
TCP Session Hijacking
This was the most complex task. An active Telnet session's sequence and acknowledgement numbers were read from Wireshark (with relative sequence numbers disabled). The command mkdir jason was converted to hex and injected as a forged TCP packet using Netwag Tool 36, spoofing the server's source IP and port. The Client's Wireshark showed the forged telnet data arriving, and running ls on the client confirmed a new directory had been created — demonstrating a full TCP session hijack.
Netwag (Tool 36) Wireshark Telnet Session Hijacking
Week 6 · Lab 5

SNORT
IDS / IPS

Snort was introduced as both a network Intrusion Detection System (IDS) and an Intrusion Prevention System (IPS). Custom rules were written and tested against real attack traffic generated from the Attacker VM.

// Task 1–3
Writing Custom Snort Detection Rules
Three detection rules were added to /etc/snort/rules/local.rules: a rule to alert on any ICMP packet, a rule to alert on web access (TCP port 80), and a rule to specifically detect ICMP Source Quench packets (type 4, code 0). Alerts were verified by reading /var/log/snort/alert and by running Snort in live console mode. Each rule used a unique SID above 1000000.
Snort Snort Rules (SPL) ICMP / TCP IDS Mode
// Task 4–5
Snort as IPS — Rejecting SSH & Telnet
Using the reject action (instead of alert), Snort was configured to actively block SSH connection attempts to port 22 and Telnet attempts to port 23. When the Attacker VM tried to connect, it received "Connection reset by peer" — the connection was terminated by Snort sending a TCP RST before it could be established. Live console output confirmed the rejection events as they happened.
Snort IPS Mode TCP Reject SSH / Telnet
// Task 6
Juniper SRX IDS — UDP Flood Threshold
Connected to a real Juniper SRX router via SSH through the UTS boundary router. Inside the edit security screen ids-option untrust-screen hierarchy, a UDP flood threshold of 1024 packets/second was set. The show command confirmed the new UDP flood configuration appeared under the existing TCP syn-flood and land attack protections. The rule was then deleted to restore default config.
Juniper SRX JUNOS CLI IDS Screen
// example snort rule structure used in lab
# Alert rule (IDS) — detect ICMP
alert icmp any any -> any any (msg:"ICMP Packet found"; sid:1000001; rev:1;)

# Reject rule (IPS) — block SSH connection attempts
reject tcp any any -> 10.0.2.6 22 (msg:"SSH Connection Attempt"; sid:1000004; rev:1;)

# ICMP type/code specific detection
alert icmp any any -> any any (msg:"Source Quench"; itype:4; icode:0; sid:1000003; rev:1;)
Week 7 · Lab 6

JUNIPER SRX
SECURITY POLICIES

Building on the Week 6 Juniper introduction, this lab focused on configuring zone-based security policies on the Juniper SRX to control traffic between the Internet zone and the Internal zone.

// Task 1
Exploring Default Security Policies
After rolling back to rescue configuration, the existing policies were inspected using show security policies. The default setup had two permit-all policies: one for Internet→Internal traffic and one for Internal→Internet traffic, both matching any source, destination, and application. This "allow everything" default was the baseline to understand before adding more specific controls.
JUNOS CLI Juniper SRX Security Zones
// Task 2
Configuring FTP Access Policy
New security policies were added for both directions (Internet→Internal and Internal→Internet) using the application junos-ftp to permit only FTP traffic. The policy structure followed the template: from-zone, to-zone, policy name, match criteria (source/destination/application), and then action (permit). This created a controlled whitelist for a specific application.
JUNOS CLI junos-ftp Zone Policy
// Task 3 (Challenge)
Rejecting Telnet Access
The challenge required creating policies using junos-telnet with a reject action for both directions between zones. This prevents Telnet traffic from traversing the firewall in either direction, while the existing permit-all policy would still allow other traffic types.
JUNOS CLI junos-telnet Reject Action
Week 9 · Lab 7

IPTABLES &
FIREWALL AUTH

The final labs covered Linux firewall configuration with iptables and enterprise-grade user authentication on the Juniper SRX using pass-through authentication.

// iptables Challenge
Permitting HTTP/HTTPS, Dropping All Other Traffic
Three iptables rules were added to the INPUT chain on Cybersec-Server: ACCEPT TCP port 80 for NEW and ESTABLISHED connections, ACCEPT TCP port 443 for NEW and ESTABLISHED connections, and a final DROP rule for all remaining traffic. The ordered chain is critical — iptables processes rules top-to-bottom, so specific ACCEPT rules must come before the catch-all DROP. Running sudo iptables -L confirmed the chain was configured correctly.
iptables HTTP (80) HTTPS (443) DROP rule
// JUNOS — Firewall User Auth
Pass-Through Authentication on Juniper SRX
An access profile was created with a named client and firewall-user password. This profile was associated with pass-through authentication in the access firewall-authentication hierarchy, with a success banner configured for Telnet. A security policy was then created to match junos-telnet traffic with a permit action that required firewall-authentication pass-through client-match — meaning users must authenticate before the Telnet session is allowed through.
Juniper SRX JUNOS CLI Pass-Through Auth Access Profiles

05 · Lessons Learned

SKILLS
LEARNED

Across the semester, these labs built a layered understanding of cybersecurity from the ground up — connecting theory to hands-on practice with real tools.

01
Packet Analysis — Reading and interpreting raw network traffic in Wireshark to find credentials, images, and protocol-level attack patterns across TCP, UDP, ICMP, DNS, and Telnet.
02
Offensive Techniques — Understanding how SQL injection, password cracking, DNS spoofing, TCP session hijacking, and ICMP attacks actually work at a technical level — not just theoretically.
03
Cryptography in Practice — Applying AES and RSA encryption, understanding the difference between cipher modes (ECB vs CBC), and building a complete PKI chain from scratch with OpenSSL.
04
Intrusion Detection & Prevention — Writing Snort rules from scratch, understanding the difference between IDS (alert) and IPS (reject), and tuning rules for specific protocols and ICMP codes.
05
Enterprise Firewall Config — Navigating the JUNOS CLI (shell, operational, and configuration modes), configuring security zones, policies, IDS screens, and firewall user authentication on real Juniper SRX hardware.
06
Linux Security Tooling — Using iptables to build stateful firewall rules on Linux, understanding rule ordering, and applying the principle of least privilege to network access control.
07
Password Security — Seeing firsthand how quickly weak and dictionary-based passwords fall to automated cracking tools, reinforcing the importance of long, random, non-dictionary passwords.
08
Threat Mindset — Thinking from the attacker's perspective first — understanding how vulnerabilities are discovered and exploited — to then reason about what defences are actually effective.
// Final Reflection

The biggest takeaway from these labs is that most attacks exploit well-known, preventable weaknesses — unencrypted protocols, weak passwords, unvalidated inputs, and misconfigured services. Cybersecurity is less about exotic zero-days and more about getting the fundamentals right. Tools like Wireshark and Snort make the invisible visible — and understanding how attacks look in packet captures is invaluable for anyone working in a defensive security role.
